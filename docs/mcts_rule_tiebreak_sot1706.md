# SOT-1706 MCTS 発動拡大・ルール tie-break 評価

## 結論

候補構成は **不採用** とした。temperature 0.15、MCTS 0.6 秒、重要局面トリガー拡大、
ルール tie-break の組み合わせを 400 戦で評価した結果、vs RuleAgent は 219 勝 181 敗
（勝率 0.5475、Wilson 95% CI `[0.4985, 0.5956]`）だった。点推定は現行の 0.540 を
上回ったが、CI 下限が 0.5 を超えないため、Issue の昇格条件を満たさない。

したがって `agents/`、`main.py`、`train/`、`data/policy.json` の提出挙動は変更していない。
以下の実験 JSON と敗因計測ハーネスだけを保存する。

## 敗因プロファイル

現行構成（temperature 0.25、MCTS 0.4 秒）を RuleAgent と 100 戦（seed 170600）対戦させ、
RESULT trace の engine end reason を集計した。

| 敗因 | 件数 | 敗戦中の割合 |
| --- | ---: | ---: |
| no_active | 47 | 94% |
| prize_out | 3 | 6% |
| deck_out | 0 | 0% |
| card_effect | 0 | 0% |

全 100 戦で 50 敗、fault 0。主因は山札切れではなく、後続の Active を用意できない
`no_active` だった。この結果を踏まえ、終盤・lethal/被 lethal・プライズ選択・方策 top2
拮抗時に探索を広げ、拮抗時は確定 KO、攻撃者へのエネルギー集約、盤面を進めない選択の
抑制を tie-break として試した。候補構成は昇格しなかったため、最終提出構成の敗因分布は
変更前と同一（コード同一）である。

生データ: `eval/sot1706_loss_profile_before.json`。再計測は次で行える。

```bash
venv/bin/python eval/loss_profile.py --n 100 --seed 170600 \
  --json eval/sot1706_loss_profile_before.json
```

## Probe

各 30 戦、fault 0、違法候補 0、1 手 5 秒上限で比較した。

| temperature | MCTS 予算 | 勝率 | Wilson 95% CI | MCTS 発動率 | 最大思考時間 |
| ---: | ---: | ---: | --- | ---: | ---: |
| 0.15 | 0.6 s | 0.600 | `[0.4232, 0.7541]` | 0.728 | 613 ms |
| 0.25 | 0.8 s | 0.533 | `[0.3614, 0.6977]` | 0.724 | 814 ms |
| 0.35 | 0.6 s | 0.533 | `[0.3614, 0.6977]` | 0.714 | 614 ms |

現行 MCTS 発動率 0.259（既存 N=400 記録）に対し候補構成は約 0.71–0.73。予算はいずれも
5 秒上限内だが、0.8 秒は勝率上の利点が見えず、0.15 / 0.6 秒を本評価へ進めた。

## N=400 昇格判定

100 戦ずつ seed 170700 / 170800 / 170900 / 171000 に分割して集計した。

| 指標 | 結果 |
| --- | ---: |
| 勝敗 | 219–181 |
| 勝率 | 0.5475 |
| Wilson 95% CI | `[0.4985, 0.5956]` |
| fault / 違法候補 | 0 / 0 |
| MCTS 発動 | 4,489 / 6,275（0.7154） |
| MCTS override | 478 |
| rule tie-break | 30 |
| 最大思考時間 | 620 ms（上限 5,000 ms） |

チャンクは `eval/sot1706_rule_chunk_{0,1,2,3}.json`、集計は
`eval/sot1706_rule_n400.json` に保存した。CI 下限 0.4985 ≤ 0.5 のため不昇格とし、
一時実装したトリガー拡大・tie-break・temperature/予算変更は revert した。

## Random 非劣化と安全性

最終成果物は現行提出コードと同一であるため、提出挙動に回帰はない。現行構成の既存
vs Random N=100（seed 2100）は 79 勝 21 敗、fault 0、最大 402 ms だった。候補構成は
RuleAgent 昇格ゲートで不採用になったため、候補を提出既定値へ反映していない。

## 受け入れ条件との対応

- 敗因分布: N=100 の trace 集計を記録。
- 発動率: 現行 0.259 に対し候補 N=400 は 0.7154。
- 調整: トリガー拡大、rule tie-break、temperature/予算を probe と本評価で検証。
- 昇格判定: N=400 を実施し、不採用を決定。
- 安全性: 全候補ベンチ fault 0、違法候補 0、最大 620 ms < 5 秒。
- 非昇格時処理: 挙動変更を revert。`train/` と `data/policy.json` は無変更。
